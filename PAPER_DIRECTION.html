Paper direction — CI2Lab as a research contribution

Strategic synthesis for turning CI2Lab into a publishable paper. Combines a state-of-the-art survey of local/on-device agent harnesses with a grounded, code-cited audit of what CI2Lab actually implements. All repo claims below are backed by file/line evidence from four subsystem audits.

One-line recommendation: Build the paper around the benchmark harness that already exists in the repo, framed as a harness-isolation study on local models, with an evidence-derived failure taxonomy as the conceptual spine. Do not headline governance, multi-agent orchestration, or “another local harness” — the audits show those are derivative or overclaimed.

1. The strategic picture

The state of the art in local/on-device agents already knows how to do local planners (CAMPHOR, Octo-planner), local tool callers (AgentFlux, TinyAgent, Hammer), local context management (Adaptive Context Management), and local runtimes (Ollama benchmarking, ARM/RISC-V quantization, speculative decoding). What it does not do well:

  • Isolate how much of local-agent reliability comes from the harness vs. the model — every SOTA paper conflates the two.
  • Measure agents beyond task success: provenance, evidence, false-positive success, failure attribution, cost-per-success.
  • Treat operational reliability (deterministic task closure, failure classification) as a measurable object rather than an engineering detail.

CI2Lab’s differentiator is therefore not any single mechanism. It is that CI2Lab can measure harness-level reliability on a fixed local model — because the benchmark scaffolding is already built.

3. Recommended thesis (primary paper)

“The harness, not just the model: isolating harness-level reliability for small local LLM agents.”

Research question. On a fixed local model, how much of task reliability is determined by the harness rather than the model? SOTA local-agent work conflates model and harness; CI2Lab can hold the model constant (one Ollama-served 7B) and vary the harness, because the benchmark already registers ci2lab, ci2lab-multi, claude-code, and codex as conditions.

Three contributions, all backed by built components:

  1. A reliability benchmark for local agents measuring beyond task success — false-positive rate (claimed-but-unverified success), evidence/provenance, failure attribution, cost-per-success.
  2. An evidence-derived failure taxonomy separating model vs harness vs tool/environment faults, including the novel tool_trace_failed (outcome correct, provenance unprovable).
  3. Empirical evidence that deterministic, model-external harness mechanisms materially lift weak-local-model reliability — via ablations (guardrails on/off, evidence-grounded closure vs LLM-judge) and the head-to-head harness comparison.

Why it survives review: the contribution is the measurement and finding, so the “it’s just engineering” critique doesn’t bite; it uses only robust, built components; and harness-isolation on local models is a genuine literature gap.

4. What to claim — and what NOT to

Claim: - Harness-isolated reliability measurement on a fixed local model. - The model/harness/environment failure taxonomy (incl. tool_trace_failed). - Deterministic evidence-grounded closure — scoped honestly as the --multi-agent / opt-in path.

Do NOT claim (each would sink credibility): - “Novel governance/security” — it is OpenCode re-implemented, no threat model. - “Secret protection” — filename-only blocking, no value redaction. - “General artifact validation” — no compile/test-based check exists. - Default-path deterministic closure — the default is LLM-judge + heuristic. - Multi-agent superiority over CAMPHOR — not demonstrated.

5. Optional second paper (only with more investment)

Tool churn / Yard gateway. The list/describe/run O(1)-schema pattern (ci2lab/harness/tools/yard_tool.py) is the cleanest systems idea and directly targets AgentFlux’s retrain-per-tool weakness. But it is O(1) only for Yard, not MCP (every MCP tool schema is injected every turn). To make it a sharp standalone paper you would need to (a) generalize progressive disclosure to MCP, and (b) show it scales to many tools without accuracy loss. Otherwise keep it as one section of the primary paper.

6. Roadmap — P0 fixes before any results

  1. Fix the telemetry bug. The bench adapter reads run.json, but the single-agent loop writes run_summary.json, so rounds / failure-attribution are silently dropped for the primary condition (ci2lab/bench/adapters/ci2lab_adapter.py). Also map the "stuck" status (absent from _RUN_STATUS_MAP).
  2. Run the live matrix once. Nothing is committed end-to-end yet; do a shakeout on a 7B model at k=5 and commit a results.jsonl.
  3. Add bootstrap CIs to summary.json (_aggregate currently emits point estimates only; methodology calls for CIs).
  4. Author harder tasks. 12 fixtures won’t carry a conference paper; expand the multi-step / recovery / safety set.
  5. Fix the stale orchestrator docstring (orchestrator.py:1-6) so reviewers don’t catch the contradiction.

Known data-integrity caveats to respect in the study: - benchmark_score in catalog/models.json is a hand-authored prior, not measured — never cite it as a result. - Runs are auditable but not replayable (no seed capture / no replay harness).

7. Related work map (for the paper’s SOTA section)

Agent architectures / harnesses (closest neighbours): - CAMPHOR — hierarchical on-device multi-agent; happy-path only, weak multi-turn. - Octo-planner — planner/action split; plans upfront, no step-wise replanning. - AgentFlux — decoupled tool selection/args; retrain per new tool (the gap). - Efficient On-Device Agents via Adaptive Context Management — compressed context state, just-in-time schemas; no permissions/longitudinal eval.

Tool calling / robustness: - TinyAgent — ToolRAG retrieval + quantization for edge function calling. - Hammer — irrelevance augmentation / function masking for generalization. - Utility-Guided Agent Orchestration — cost/quality decision policy. - ReST meets ReAct — self-improvement + distillation of a ReAct controller.

Failure-mode / self-correction literature (motivation): - AgentBench and “Where LLM Agents Fail…” — document loops / budget exhaustion. - Reflexion, Self-Refine, ReflexGrad — reflection-based self-correction. - ICLR 2024 intrinsic self-correction result — LLMs cannot reliably self-correct unaided (key: motivates model-external mechanisms for weak models).

Runtime enablers: - LLMs at the Edge with Ollama; Latency-Critical Quantized Inference on ARM/RISC-V; Memory-Efficient Speculative Decoding on commodity GPUs; vla.cpp.

Benchmarks: MCP-Bench (28 servers, 250 tools) — closest tool-use benchmark, but does not cover persistent/governable/auditable local-agent behavior.
